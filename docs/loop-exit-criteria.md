# Loop 종료 조건 (Exit Criteria)

fg-loop 무인 주행의 정지 조건을 정의한다. **모든 판정은 명령 실행과 종료 코드만으로 이루어진다** — 사람 판단이 필요한 항목은 종료 조건이 될 수 없으며 §4의 "벽"으로 분리한다. 기능 목록의 원천은 [PRD](./PRD.md), 검증 명령의 스크립트 이름은 이 문서가 원천이다.

## 1. 판정 규칙

> **아래 검증 명령이 클린 체크아웃에서 순서대로 전부 종료 코드 0으로 끝나면 loop는 종료된다. 하나라도 0이 아니면 미완료다.**

- 부분 성공, "거의 됨", 로그 육안 확인은 판정에 쓰지 않는다.
- 테스트를 skip/disable 처리해 통과시키는 것은 종료로 인정하지 않는다 — E2E 매핑 표(§3)의 시나리오가 **존재하고 실행되어 통과**해야 한다.
- 기능이 PRD에 추가되면 매핑 표에 행이 먼저 추가되어야 한다(매핑 없는 기능 = 검증 불가 = 머지 불가).

```
pnpm verify 실행
   ├─ 전부 exit 0 → loop 종료 (목표 달성)
   └─ 하나라도 실패 → 실패 지점 수리(fix-forward) → 재실행
```

## 2. 검증 명령

루트 `package.json`에 아래 스크립트가 있어야 하며, `pnpm verify`가 전부를 순서대로 실행한다.

| 순서 | 명령 | 검증 내용 | 전제 |
|------|------|----------|------|
| 1 | `pnpm typecheck` | backend·frontend·shared 전체 `tsc --noEmit` | `pnpm install` |
| 2 | `pnpm lint` | ESLint 전 워크스페이스 | — |
| 3 | `pnpm build` | backend·frontend 프로덕션 빌드 | — |
| 4 | `pnpm --filter backend test:e2e` | NestJS 통합 테스트(supertest) — API 계약·워크스페이스 격리 | 테스트용 Postgres (docker compose) |
| 5 | `pnpm test:e2e` | Playwright — §3 매핑 표 전 시나리오 | backend+frontend 기동(Playwright webServer), 테스트 DB |

- 4·5번은 격리된 테스트 DB를 스스로 준비(마이그레이션 포함)해야 한다 — 사람이 미리 시드하는 전제는 금지.
- E2E는 headless로 돌고 CI 환경 변수 없이 로컬에서 같은 명령으로 재현된다.

## 3. 기능 ↔ E2E 시나리오 매핑

PRD의 기능 하나는 아래 스펙 파일의 시나리오 하나 이상으로 매핑된다. **이 표의 모든 행이 구현·통과되어야 종료다.**

| PRD 기능 | 스펙 파일 (`e2e/`) | 필수 시나리오 |
|----------|-------------------|--------------|
| F1 인증 | `auth.spec.ts` | 가입→자동 워크스페이스 진입 / 로그아웃→보호 경로 차단 / 타 계정 페이지 API 접근 거부(403·404) |
| F2 페이지 트리 | `pages-tree.spec.ts` | 문서·DB 페이지 생성 / 하위 페이지 중첩(3단 이상) / 이름 변경이 트리 반영 / 이동(부모·순서 변경) / 새로고침 후 트리 유지 |
| F3 문서 편집 | `editor.spec.ts` | 슬래시 메뉴로 헤딩·리스트·코드 블록 생성 / 서식(굵게·링크) / 자동 저장 후 새로고침 복원 / 제목 편집이 사이드바 반영 |
| F4 데이터베이스 | `database.spec.ts` | 속성 6종 각각 추가·값 편집 / 속성 이름·타입 변경, 삭제 / 행 추가·삭제 / 행 열기→속성 편집 영역+본문 편집 |
| F5 뷰 3종 | `database-views.spec.ts` | 테이블 셀 인라인 편집 / 보드: 셀렉트 기준 그룹핑+카드 드래그로 값 변경 / 리스트 표시 / 뷰 추가·전환·삭제 / 필터·정렬 적용이 뷰별로 저장 |
| F6 전역 검색 | `search.spec.ts` | 제목 매칭 / 본문 저장 직후 본문 텍스트 매칭(한국어 부분 문자열) / 결과 클릭 이동 / 휴지통 페이지 제외 |
| F7 휴지통 | `trash.spec.ts` | 삭제 시 하위 포함 트리에서 소거 / 복원(부모 삭제 시 최상위 복원 포함) / 영구 삭제 후 미복원 확인 |
| F8 즐겨찾기 | `favorites.spec.ts` | 토글 on→섹션 표시·클릭 이동 / off→소거 / 삭제된 페이지 섹션에서 소거 |
| F9 아이콘·커버 | `page-decor.spec.ts` | 이모지 설정→트리·헤더 표시·제거 / 프리셋 커버 설정·변경·제거 |
| F10 다크모드 | `theme.spec.ts` | 토글 전환 / 새로고침 후 유지 / 에디터·DB 뷰 영역 테마 적용 확인 |

백엔드 통합 테스트(supertest)는 최소한 다음을 덮는다: 전 엔드포인트([architecture.md](./architecture.md) §4)의 정상 경로, 워크스페이스 격리(타 계정 자원 접근 거부), 휴지통 전파·복원 규칙.

## 4. fg-loop가 멈춰야 하는 벽 (기계 판정 불가 항목)

다음은 exit 0으로 판정할 수 없으므로 **종료 조건에 넣지 않는다**. loop 중 이 판단이 필요해지면 멈추고 사람에게 넘긴다.

- **시각 품질·UX 완성도** — "노션처럼 보이는가"는 사람 판단. E2E는 동작만 검증한다.
- **성능 체감** — v1 종료 조건에서 제외(측정 기준을 세우는 것 자체가 별도 태스크).
- **PRD 범위 변경** — 기능 추가·축소는 재그릴링(fg-ask) 대상이지 loop가 스스로 결정하지 않는다.
- **설계 분기** — 두 갈래 이상의 구현 방향에서 문서(PRD·architecture)로 판정 불가한 선택.

## 5. fg-loop 설정 시 권장 값

- **정지 조건**: `pnpm verify` 종료 코드 0 (§1·§2 그대로).
- **fix-forward 허용 범위**: PRD·architecture 문서 안에서의 구현·수리. 문서와 충돌하는 수리는 벽(§4)으로 취급.
- **재계획 상한**: 기본값 3회 유지.
